Element 5 Documentation – Realm Switching

This documentation explains the design and implementation of Scene 5, the final BabylonJS element in the Mindspace practical portfolio. Scene 5 is a realm-switching demo inspired by The Withering of the World Tree, where the player travels between Midgard, the Yggdrasil realm hub, the remnants of Asgard, and Muspelheim.

The main purpose of this scene is to demonstrate scene switching using GUI controls and in-world interactions, while also presenting the result as a small playable game sequence rather than a simple technical test.

Scene 5 concept

Scene 5 is structured around Yggdrasil as a central realm-travel hub. The player can begin from a realm-selection menu, enter Midgard, activate a portal, travel to Yggdrasil, visit Asgard to recover the shard of Mjolnir, return to Yggdrasil, and then unlock Muspelheim as the final realm.

Each realm is separated into its own HTML and JavaScript file. This made the scene easier to organise because each location has its own environment, interaction logic, UI, assets, lighting, and visual style.

Scene files and structure

Scene 5 is split across a menu page and four separate realm pages. The menu page is used to choose a starting point, while the realm pages contain the playable areas.

Interaction flow

  1. The player opens Scene 5 and sees a Yggdrasil-style realm menu.
  2. The player can start the intended route by selecting Midgard.
  3. In Midgard, the player follows the ruined road to the stone temple.
  4. The player interacts with the temple portal to travel to the Yggdrasil hub.
  5. In Yggdrasil, the player can inspect realm gates and travel to Asgard.
  6. In Asgard, the player collects the shard of Mjolnir.
  7. The player returns to Yggdrasil and uses the shard to unlock Muspelheim.
  8. In Muspelheim, the player reaches Surtr’s Forge and completes the realm route.

Scene switching implementation

Scene switching is handled using separate HTML pages for each realm. GUI buttons and in-world interactables change the page using window.location.href. This means each realm can load its own BabylonJS scene independently while still feeling like part of one connected route. 

addInteractable("Enter the Yggdrasil Realm Travel Room", new BABYLON.Vector3(0, 0, 21.1), 3.0, () => {
    window.location.href = "yggdrasil.html";
});

The main Scene 5 menu uses Babylon GUI buttons to send the player to the selected realm page. 

button.onPointerUpObservable.add(() => {
    window.location.href = href;
});

Progress and realm unlocks

Some progress is stored with localStorage. This is used so the Yggdrasil hub can remember whether the player has collected the shard of Mjolnir or unlocked Muspelheim, even after moving between different realm pages. 

const progress = {
    hasShard: localStorage.getItem("scene5HasMjolnirShard") === "true",
    muspelUnlocked: localStorage.getItem("scene5MuspelheimUnlocked") === "true"
};

When the player collects the Mjolnir shard in Asgard, the value is saved. Yggdrasil can then read this value and allow the player to unlock the Muspelheim gate. 

localStorage.setItem("scene5HasMjolnirShard", "true");

Player movement and camera

The realm scenes use a third-person player controller. The player can move with WASD, sprint with Shift, jump with Space, interact with E, and toggle a debug camera with N. The debug camera was useful during development because it allowed each realm to be inspected from above while checking layout, collision, and visual design. 

if (keys.w) move.addInPlace(forward);
if (keys.s) move.subtractInPlace(forward);
if (keys.a) move.subtractInPlace(right);
if (keys.d) move.addInPlace(right);

Gravity and jumping are handled manually by applying vertical velocity to the player and then snapping them back to the current walk height when they land. 

player.verticalVelocity -= player.gravity * delta;
player.root.position.y += player.verticalVelocity * delta;

if (player.root.position.y <= walkHeight) {
    player.root.position.y = walkHeight;
    player.verticalVelocity = 0;
    player.isGrounded = true;
}

Player model and animation

The final Scene 5 player uses the Solus knight model, reused from Scene 2. This gave the realm sequence a more complete game-like feel than the earlier placeholder black character mesh. The Solus model is imported with BabylonJS and its animation groups are searched for idle, walk, and run animations. 

BABYLON.SceneLoader.ImportMeshAsync(
    "",
    "../../assets/",
    "Solus_the_knight.gltf",
    scene
).then((result) => {
    playerRef.idleAnim = chooseBestAnimation(result.animationGroups, ["idle"]);
    playerRef.walkAnim = chooseBestAnimation(result.animationGroups, ["walk", "move"]);
    playerRef.runAnim = chooseBestAnimation(result.animationGroups, ["run", "sprint"]);
});

Animation playback is linked to movement. If the player is not moving, the idle animation plays. If the player is walking, the walk animation plays. If the player is sprinting, the run animation plays.

Collision and interaction

Collision in Scene 5 is handled using simple custom collision zones. These are stored as circles or boxes in a blockers array. This approach was chosen because many objects in Scene 5 are decorative meshes, and simple invisible collision zones were easier to control than using full mesh collisions for everything. 

function addCollisionCircle(x, z, radius) {
    blockers.push({
        type: "circle",
        x,
        z,
        radius
    });
}

Interactions are handled using an interactables array. Each interactable stores prompt text, a position, a radius, and an action. When the player is close enough, the UI prompt appears and pressing E triggers the action. 

addInteractable("Yggdrasil Return Gate", new BABYLON.Vector3(0, 0, -10.9), 3.1, () => {
    window.location.href = "yggdrasil.html";
});

Realm design summary

Midgard

Midgard is designed as a ruined outdoor route. It includes a forest, mountain boundary, ruined structures, camp details, a cave entrance, a dirt road, and a stone temple portal. It acts as the starting point and introduces the player to the realm-travel idea.

Yggdrasil Hub

Yggdrasil is the central travel hub. It contains a large world tree, realm gates, a circular platform, portal effects, floating debris, and a shard socket used to unlock Muspelheim. It connects the different realm pages together.

Asgard Remnants

Asgard is presented as a broken floating remnant after Ragnarök. It includes a fractured stone platform, a broken throne area, floating debris, a return portal, and the shard of Mjolnir. The shard is the key item needed to progress.

Muspelheim

Muspelheim is the final volcanic realm. It contains lava streams, black basalt paths, volcanic rocks, a sword field, ember particles, and Surtr’s Forge. The player crosses the arena and interacts with the forge to complete the route.

Assets used

The scene uses a mixture of procedural BabylonJS geometry and imported assets. Many objects, such as platforms, portals, trees, ruins, gates, lava paths, collision zones, and decorative debris, are built directly in JavaScript using BabylonJS mesh builders.

User interface

The user interface is split between HTML and Babylon GUI. Each page has a top-left HTML information panel explaining the current realm. Babylon GUI is then used for in-scene prompts, objectives, controls, and the Scene 5 menu buttons.

The UI was adjusted near the end of development so the realm pages were more consistent: prompts appear near the bottom of the screen, objective text appears below the prompt, and controls remain visible without covering the main environment.

Testing and debugging

Scene 5 was tested by moving through the complete route and checking that each page loaded, player movement worked, interaction prompts appeared, and scene switching sent the player to the correct realm page. The browser console was also used to catch JavaScript errors, missing functions, incorrect paths, and missing assets.

  • Checked that the Scene 5 menu buttons open the correct realm pages.
  • Checked that Midgard can switch to Yggdrasil through the temple portal.
  • Checked that Asgard saves the Mjolnir shard state.
  • Checked that Yggdrasil reads progress and unlocks Muspelheim.
  • Checked that Muspelheim can return to Yggdrasil.
  • Used the debug camera to inspect collision, layout, scale, and visual design.

Reflection

Scene 5 developed from a simple realm-switching idea into a more complete playable sequence. The main challenge was keeping the scene manageable while still making each realm feel visually different. Splitting each realm into its own page and JavaScript file made the project easier to organise and reduced the complexity of switching between large environments.

The project helped develop a better understanding of BabylonJS scene creation, imported models, animation groups, procedural mesh building, GUI controls, localStorage progression, skybox loading, and custom interaction systems. The final Scene 5 demonstrates both the required technical scene switching and a more creative game-like structure.
